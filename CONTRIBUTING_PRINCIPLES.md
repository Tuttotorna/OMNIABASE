# OMNIABASE — CONTRIBUTING PRINCIPLES

## Status

This document defines the contribution rules for the `OMNIABASE` umbrella repository.

Its purpose is not to regulate branch-specific code style or repository-local workflows.

Its purpose is to preserve the conceptual clarity, architectural coherence, and structural boundaries of the Omniabase framework at the general level.

---

## 1. First rule

Contributions to the umbrella repository must strengthen clarity, not increase volume.

The goal of `OMNIABASE` is not to accumulate everything related to the ecosystem.

Its goal is to remain the clean architectural center of the framework.

A contribution is useful only if it makes the framework:

- clearer
- better bounded
- better structured
- easier to read
- easier to distinguish from its branches and satellites

---

## 2. Repository role must remain stable

`OMNIABASE` is the umbrella repository.

It is not:

- the main code repository of every branch
- the archive of all experiments
- the place for branch-local benchmarks
- the place for historical accumulation
- the place for speculative overflow

Contributions must respect this role.

If material belongs to a branch repository, it should remain there.

If material belongs to a satellite repository, it should remain there.

---

## 3. Framework-level language only

Contributions to this repository should use framework-level language.

This means:

- define principles
- define architecture
- define branch taxonomy
- define shared vocabulary
- define repository roles
- define stable conceptual boundaries

Avoid branch-local technical detail unless it is necessary to explain the branch structurally.

Avoid implementation-specific language unless the document explicitly concerns implementation strategy.

---

## 4. No overclaim

The umbrella repository must remain more precise than ambitious.

Contributions must not present Omniabase as:

- already universally validated
- already complete
- already able to measure everything
- already externally proven in all domains
- already reducible to one branch
- already identical to one implementation

The framework may remain expansive in direction, but public statements in the umbrella repository must remain structurally defensible.

---

## 5. Branch distinction is mandatory

Every contribution must preserve the distinction between:

- the general framework
- the canonical branches
- branch implementations
- epistemic pre-layers
- satellite repositories

No contribution should blur these roles.

In particular:

- `OMNIABASE` must not be collapsed into `OMNIA`
- Diagnostics must not be treated as the whole framework
- Coordinate Discovery must not be treated as a mere byproduct
- Translation must not be erased because it is less mature
- epistemic gateways must not be confused with the architectural center

---

## 6. Measurement, interpretation, decision

All contributions must preserve the non-negotiable separation between:

- measurement
- interpretation
- decision

A document in the umbrella repository may discuss all three layers, but it must not collapse them into one undifferentiated act.

This separation is a core structural rule of the framework.

---

## 7. Terminology discipline

Contributions should prefer the shared Omniabase lexicon where possible.

When a new term is proposed, it should satisfy all of the following:

- it names something structurally necessary
- it is not redundant with an existing term
- it clarifies rather than mystifies
- it can remain useful across branches
- it does not import unnecessary branch-specific baggage into the umbrella layer

If a term belongs only to one branch, it should remain documented in that branch context rather than elevated immediately to framework-level vocabulary.

---

## 8. Controlled branch growth

Not every new idea becomes a canonical branch.

A new branch should be recognized only if it has:

- a distinct central question
- distinct structural outputs
- a coherent methodological identity
- non-trivial difference from the existing branches
- enough scope to justify independent recognition

Until then, new work should be treated as:

- a subdirection inside an existing branch
- a branch experiment
- a satellite exploration
- an epistemic note

This prevents inflation of the framework.

---

## 9. Historical material must remain bounded

Historical relevance does not automatically imply framework centrality.

Contributions must distinguish between:

- what helped generate Omniabase
- what currently defines Omniabase
- what remains adjacent but external

The umbrella repository should preserve lineage where necessary, but should not become a full historical archive.

Its priority is architectural legibility in the present.

---

## 10. External readability matters

A reader encountering `OMNIABASE` for the first time should be able to understand:

- the core principle
- the canonical branches
- the ecosystem structure
- the role of the repository
- the conceptual limits of the framework

Contributions that only make sense to insiders without adding structural clarity should be revised, relocated, or excluded.

The framework must remain legible beyond its development history.

---

## 11. Preferred contribution types

The following contribution types are strongly aligned with the role of the umbrella repository:

- foundational clarifications
- architectural refinements
- branch taxonomy refinements
- lexicon improvements
- ecosystem mapping updates
- framework boundary clarifications
- stable roadmap updates
- reference links to branch repositories

These are preferred over bulk additions of local experiments or narrow claims.

---

## 12. Contribution test

Before adding material to `OMNIABASE`, the following questions should be asked.

### A. Does this clarify the general framework?
If not, it may belong elsewhere.

### B. Does this belong to the umbrella level rather than to a branch?
If not, it should be moved to a branch repository.

### C. Does this preserve the existing architectural distinctions?
If not, it should be rewritten.

### D. Does this make the framework easier to read for a serious outsider?
If not, it should be revised.

### E. Is this a stable contribution rather than a temporary branch-local note?
If not, it should remain outside the umbrella repository.

Only contributions that pass this test should enter the core repository.

---

## 13. Style principle

The umbrella repository should prefer:

- clarity over density
- structure over rhetoric
- boundary over inflation
- readable generality over vague universality
- stable taxonomy over repository sprawl

Omniabase should appear broader than any one branch, but tighter than the total history of its emergence.

---

## 14. Summary

The contribution rule for `OMNIABASE` is simple:

contribute only what strengthens the framework as framework.

The umbrella repository exists to remain:

- conceptually clear
- architecturally stable
- branch-aware
- ecosystem-readable
- externally legible

Anything else belongs downstream.
