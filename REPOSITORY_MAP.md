# OMNIABASE — REPOSITORY MAP

## Status

This document maps the current Omniabase ecosystem at the repository level.

Its purpose is not to rank repositories by importance, but to clarify their structural role inside the broader Omniabase framework.

The ecosystem must be read as an architecture, not as a flat list.

---

## 1. Core architectural rule

Not every repository in the ecosystem has the same role.

A repository may function as one of the following:

- umbrella framework repository
- canonical branch repository
- epistemic pre-layer
- satellite repository
- historical or adjacent repository

These roles must remain distinct.

The ecosystem becomes legible only when repositories are read by function, not by chronology or naming similarity.

---

## 2. Umbrella repository

### `OMNIABASE`

Role:

the general umbrella repository of the Omniabase framework.

Function:

- define the foundational principle
- define the common architecture
- define the canonical branches
- define the shared lexicon
- define the ecosystem map
- link to specialized repositories

This repository is not the primary place for branch-specific code, narrow benchmarks, or branch-local experimental claims.

It is the constitutional center of the framework.

---

## 3. Canonical branch repositories

The Omniabase framework currently supports three canonical branches.

Some already have repository embodiments.
Others remain partially distributed or not yet fully formalized as dedicated repositories.

---

### A. Diagnostics branch

Purpose:

study whether structure remains stable beyond a single representation.

Typical concerns:

- stability
- fragility
- drift
- collapse proximity
- post-hoc structural auditing
- structural monitoring

Primary repositories aligned with this branch:

- `OMNIA`
- `lon-mirror`
- `omnia-limit`
- `Pre-Deployment-Structural-Gate`
- `OMNIA-RADAR`

These repositories should be read as branch repositories or branch satellites of Omniabase Diagnostics.

They do not define the whole Omniabase framework.
They instantiate one of its canonical consequences.

---

### B. Coordinate Discovery branch

Purpose:

use multiple representations to expose hidden coordinates, latent factors, structural separations, and more faithful descriptive axes.

Primary repository aligned with this branch:

- `omniabase-coordinate-discovery`

This repository should be read as the clearest current embodiment of the Coordinate Discovery branch.

Future repositories may expand this branch further.

---

### C. Cross-Representation Translation branch

Purpose:

measure compatibility, alignment, and shared structural residue across different descriptions of the same object.

Current repositories aligned with this branch include:

- `omega-translator`
- `HASC-Human-AI-Structural-Compatibility-Protocol`

This branch is conceptually present in the ecosystem, but may later require a more explicitly dedicated Omniabase repository.

For now, these repositories should be read as branch-level implementations or precursors.

---

## 4. Epistemic pre-layer

Some repositories do not belong directly to a canonical branch, but prepare the conceptual transition into Omniabase.

### `observer-suspension`

Role:

epistemic pre-layer.

Function:

- weaken the monopoly of a privileged observer
- question single-view sufficiency
- prepare the shift from observer-centered description to representation-variable structural analysis

`observer-suspension` is not the whole Omniabase framework.

It is one of its strongest conceptual gateways.

---

## 5. Historical, adjacent, and satellite repositories

Some repositories are part of the broader ecosystem but should not be confused with the umbrella framework or with the canonical branch definitions.

They may still be important.

Their role may include:

These repositories may inform Omniabase without defining it.

## 7. Non-negotiable repository distinctions

The ecosystem must preserve the following distinctions.

### A. Umbrella repository != branch repository
`OMNIABASE` is not identical to Diagnostics, Coordinate Discovery, or Translation.

### B. Branch repository != satellite repository
A repository may be important without being canonical.

### C. Epistemic pre-layer != framework definition
A conceptual gateway is not the same as the umbrella framework.

### D. Historical relevance != architectural centrality
Older or influential repositories may matter historically without defining the current architecture.

### E. Naming overlap != structural equivalence
Repositories with similar names are not automatically equivalent in role.

These distinctions are required for ecosystem readability.

## 9. Future repository policy

As the ecosystem evolves, new repositories should be added only after assigning them a clear role.

A new repository should be classified as one of:

- umbrella-level document repository
- canonical branch repository
- branch implementation repository
- epistemic pre-layer
- satellite / adjacent repository

No new repository should remain structurally unclassified.

This prevents ecosystem drift.

---

## 10. Criteria for adding a repository to the map

A repository should be added to this map only if at least one of the following is true:

- it embodies one canonical branch of Omniabase
- it functions as an epistemic gateway into the framework
- it preserves historically important context for understanding the framework
- it extends the ecosystem in a way that remains structurally relevant

A repository should not be added merely because it shares vocabulary, naming, or thematic overlap.

The map must remain architectural, not exhaustive.

---

## 11. Criteria for reclassification

Repository roles are not assumed to be permanently fixed.

A repository may be reclassified if:

- its function becomes clearer over time
- a branch becomes more formally defined
- a historical satellite evolves into a branch implementation
- a branch implementation becomes obsolete or gets superseded
- the overall architecture of the framework is refined

Reclassification should remain explicit and justified.

---

## 12. Ecosystem discipline

The purpose of this map is not only descriptive.

It is also protective.

Without repository discipline, the ecosystem risks becoming:

- historically overloaded
- conceptually ambiguous
- branch-confused
- externally unreadable

This map exists to keep the framework more legible than the path that generated it.

---

## 13. Summary

The Omniabase ecosystem is not a flat collection of related repositories.

It is a layered structure.

At its center stands:

- `OMNIABASE` -> the umbrella framework repository

Around it stand:

- canonical branch repositories
- epistemic gateways
- satellites and historical layers

The purpose of this map is to preserve one rule above all:

the framework must remain more legible than its history.
